Introduction

  • Concurrency in Software execution can occur at 4 levels
    • Instruction level
    • Statement level
    • Unit or subprogram level
    • Program level
  • Instruction level and program level concurrency have nothing to do with programming language descriptor
  • Statement level and subprogram level involve design decisions while designing PLs

Multiprocessor Architecture

  • We use the phrase physical concurrency when multiple processors are actually used to execute concurrent units
  • If concurrent units are executed on a single processor in an interleaved fashion it is known as logical concurrency
  • Most multiprocessor computers either belong to SIMD or MIMD

SIMD

  • Single instruction but multiple data
  • Each processor has it’s own memory
  • Multiple processors execute same instruction simultaneously where each processor working on different data
  • One processor called controller, controls the operations of other processors
  • Scientific computations are the most suited kind
  • Widely used for graphics and video processing

MIMD

  • Computers have multiple processors that operate independently but whose operations can be synchronized are called MIMD
  • Each processor executes its own instruction stream
  • MIMD can be one of the following type:
    • Distributed - each processor has its own memory
    • Shared memory - all processors share the same memory and memory access clashes can occur

Categories of Concurrency

  • We use the phrase physical concurrency when multiple processors are actually used to execute concurrent units
  • If concurrent units are executed on a single processor in an interleaved fashion it is known as logical concurrency
  • Logical concurrency allows the programmer and application software to assume that there exist multiple processors providing actual concurrency but in fact the execution of programs happen on same processor in interleaved fashion
  • From the programmer and language designer point of view the logical concurrency is same as physical concurrency
  • It is language implementer’s task using the OS to map logical concurrency to the hardware

Motivations for Concurrency

  • Speed of execution of programs on machines with multiple processors.
  • Single processor concurrent execution can be much faster than sequential execution
  • Concurrency provides better solution for certain problems, like how recursion can help in certain situations.
  • Applications can be distributed over multiple machines locally or through the Internet

Subprogram Level Concurrency

  • Task is a unit of a program that can be in concurrent execution with other units of the same program
  • Each task in a program can support one thread of control
  • Tasks are sometimes called processes
  • Heavyweight tasks have their own address space
  • Lightweight tasks all run under same address space

Advantages of LWTs

  • Easy to implement
  • Less effort to manage their execution
  • Inter process communication is easy through non local variables and if they don’t communicate with each other they are said to be disjoint

Synchronization

Synchronization is a mechanism that controls the order in which tasks execute.

Cooperation Synchronization

  • Required between A and B when A must wait for B to finish
  • Producer Consumer Problem

Competition Synchronization

  • Required between tasks when both tasks required the use of same resource that cannot be used simultaneously
  • Prevents two tasks from modifying the same data at the same time leading to better integrity of the system

Two main features are

  • Mutually exclusive access to shared data
  • Cooperation among task

Task can be in any one of the following states: new, ready, running, blocked and dead

A. Semaphore

  • It is a data structure consisting of an integer and a task description queue where a task descriptor will describe all the information needed for the execution of the task
  • Semaphores can be used for implementing both competition and cooperation synchronizations.
  • Easy to use semaphores incorrectly resulting in errors that cannot be detected by the compiler, linker, or run-time system.

B. Monitors

  • Are data abstractions that provide a natural way of providing mutual exclusion. Supported by Java, Ada, C# etc.
  • They encapsulate shared data structures and protect their methods

C. Message passing

  • The underlying concept is to allow tasks to send messages to each other to synchronize their execution

Design Issues for Concurrency

  • How competition and cooperation synchronization are provided
  • How an application can influence task scheduling
  • How and when tasks start and end execution
  • How and when they are created
  • A program has the liveness characteristic if it continues to execute, eventually leading to completion
  • Loss of liveness and deadlock

Miscellaneous

  • API Open MP - Open Multiprocessing supports multi platform shared memory multiprocessing programming in C, C++ and Fortran → #include <omp.h>
  • Tasks in Ada are heavyweight tasks.
  • Java supports lightweight tasks relatively simple but effective and has less overhead than ada
  • Java supports monitors for implementing competition synchronization.
  • Java has – wait, notify methods for implementing cooperation synchronization.
  • Java has join, sleep, interrupt methods. Java also has Semaphore class
  • Java constructors cannot be synchronized but only methods
  • An object whose all methods are synchronized is a monitor
  • C# supports concurrency based on Java but slightly sophisticated.
  • All .NET languages have use of the concurrent data structures for stacks, queues and bags where competition synchronization is implicit.

Java Code

import java.util.concurrent.*;

class Shared{
	static int count = 0;
}

class MyThread extends Thread{
	Semaphore sem;
	String threadName;

	public MyThread(Semaphore sem, String threadName){
		super(threadName);
		this.sem = sem;
		this.threadName = threadName;
	}

	public void run(){
		if(this.getName().equals("A")){
			System.out.println("Starting " + threadName);

            try{
                // Try to get access to the resource
                System.out.println(threadName + " is waiting for a permit.");
                sem.acquire();
                System.out.println(threadName + " gets a permit.");

                // Access the resource
                for(int i = 0; i < 5; i++){
                    Shared.count++;
                    System.out.println(threadName + ": " + Shared.count);

                    Thread.sleep(10);
                }
            } catch(InterruptedException exc){
                System.out.println(exc);
            }

            // Release the permit
            System.out.println(threadName + " releases the permit.");
            sem.release();
		}
        else{
            System.out.println("Starting " + threadName);

            try{
                // Try to get access to the resource
                System.out.println(threadName + " is waiting for a permit.");
                sem.acquire();
                System.out.println(threadName + " gets a permit.");

                // Access the resource
                for(int i = 0; i < 5; i++){
                    Shared.count++;
                    System.out.println(threadName + ": " + Shared.count);

                    Thread.sleep(10);
                }
            } catch(InterruptedException exc){
                System.out.println(exc);
            }

            // Release the permit
            System.out.println(threadName + " releases the permit.");
            sem.release();
        }
	}
}

public class PoPL{
    public static void main(String[] args){
        Semaphore sem = new Semaphore(1);
        MyThread mt1 = new MyThread(sem, "A");
        MyThread mt2 = new MyThread(sem, "B");

        mt1.start();
        mt2.start();

        try{
            mt1.join();
            mt2.join();
        } catch(InterruptedException exc){
            System.out.println(exc);
        }
    }
}
