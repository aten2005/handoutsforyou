Reasons for Studying Concepts of Programming Languages

Increased capacity to express ideas

  • It’s generally harder for people to conceptualize structure they cannot describe, verbally or in writing
  • Programmers also are similarly constrained
  • Awareness of a wider variety of programming language features can reduce limitations in software development
  • A C programmer who had learned the structure and uses of associative arrays in Perl might design structures that simulate associative arrays in that language

Improved background for choosing appropriate languages

  • Most training programs limit instruction to one or two languages that are only relavant to their projects
  • Problem is that programmers will choose the language they know rather than the one that is suited for the project
  • If these programmers had access to more constructs, they would be better able to choose the language that best solves the problem

Increase ability to learn new languages

  • Once a thorough understanding of the fundamental concepts of a language is obtained, it becomes far easier to see how these concepts are incorporated into the design of the language being learned
  • Programmers who understand OOP will have much easier time learning Java

Better understanding of the significance of implementation

  • In learning the concepts of programming languages, it is both interesting and necessary to touch on the implementation issues that affect those concepts
  • In some cases understanding the implementation issues lead to an understanding of why that language was designed
  • Certain bugs can be found and fixed only by a programmer who knows some related implementation details

Better use of languages that are already known

  • By studying the concepts of programming languages, programmers can learn about previously unknown and unused parts of the languages they already use and begin to use those features.

Language Evaluation Criteria

Readability

  • The software life-cycle concept was developed and updated to give a smaller role to coding and maintenance was recognized as the large part of the cycle
  • Readability has multiple characteristics that contribute to it’s readability
  • Readability must be considered in the context of the problem definition, for example, if a program that describes the computation is written in a language not meant for such use, the program will be unnatural and convoluted making it difficult to read

Overall Simplicity

  • Language with a large number of basic constructs is more difficult to learn than the one with a smaller one
  • Programmers tend to use only a subset of these constructs
  • Feature multiplicity → having more than one way to accomplish a particular operation
    • count = count + 1;
    • count += 1;
    • count++;
    • ++count;
  • This decreases readability
  • Operator overloading
    • It is often useful but only if the overloading is done sensibly
    • For example, overloading + to add floating point numbers and integers simplifies the language but defining + as the subtraction of two numbers over complicates the language
  • Simplicity can go too far too, assembly language is supposed to be very simple but they lack more complex statements and this leads to lower readability due to the language being very simple

Orthogonality

  • This means that a relatively small set of primitive constructs can be combined in a relatively small number of ways to build the control and data structures of the language
  • Every possible combination or primitives is legal and meaningful
  • For example, in C we have a predefined set of data types but through combination of int, float, double, char and array, pointer we can define further more data structures
  • The meaning of an orthogonal language feature is independent of the context of it’s appearance in a program
  • Lack of orthogonality leads to exceptions to the rules of the language
  • Orthogonality related to simplicity. More orthogonal the language, fewer exceptions the language requires and fewer exceptions leads to a higher degree of regularity in the design, which makes the language easy to learn, read and write
  • Too much orthogonality also can cause problems
  • Most orthogonal language is ALGOL 68 → every construct has a type and there are no restrictions on those types
  • Functional languages → combination of simplicity and orthogonality

Data Types

  • Suppose a numeric type is used for an indicator flag because there is no boolean type, in such a language timeOut = 1 will be used
  • But if it exists, then timeOut = true will be used and the second statement gives a better view

Syntax Design

  • Special Words →
    • Fortran and Ada don’t have closing brackets which might not be clear as to what is ending, they have end if and end loop
    • This is an example of the conflict between C++ which has simplicity compared to Ada which has better readability
    • Another problem is whether the special words of a language can be used as names for variables, if so then the programs become very confusing
    • GREP comes from g/regularexpression/p
  • Form and Meaning
    • In C the meaning of static keyword depends on context of appearance

Writability

  • Measure of how easily a language can be used to create programs for a chose program domain
  • Writing a program requires reading it first

Simplicity and Orthogonality

  • Large number of constructs → programmers not familiar with all of them
  • Leads to misuse and disuse of others
  • Smaller number of primitive constructs and a consistent set of rules for combining them → better than simply having large number of primitives
  • Too much orthogonality can decrease writability
  • Errors in programs go undetected

Support for Abstraction

  • Ability to define and then use complicated structures or operations in ways that allow many of the details to be ignored
  • Process abstraction is the use of subprogram to implement a sort algorithm that is required multiple times in the program and if there was no subprogram we would have to replicate it in a lot of places
  • Data abstraction is where we store binary trees in the form of integer arrays, for example, in Fortran 77 as 3 parallel integer arrays where 2 of the integers represent offspring nodes

Expressivity

  • count++ more comfortable than count = count + 1
  • We have and then operator in Ada and is a convenient way of specifying short circuit evaluation of a boolean expression
  • Inclusion of for and while in java make writing loops easier

Reliability

  • Performs to specifications under all conditions

Type Checking

  • Testing for type errors in a given program by compiler or during execution
  • Run time type checking is expensive, compile time type checking is more desirable
  • Earlier the errors detected, the less expensive it is to make the required changes
  • Java checks for all types
  • In C we had issues where int could be used as a function parameter for a float variable

Exception Handling

  • Ability to intercept run time errors, take corrective measures and then continue → reliable
  • Ada, C++, Java and C# → extensive exception handling

Aliasing

  • Two or more distinct names that can be used to access the same memory cell
  • Most languages allow some kind of aliasing

Readability and Writability

  • Readability affects reliability in both the writing and maintenance phases of the life cycle

Cost

  • Cost of training programmers to use the language → simplicity and orthogonality
  • Cost of writing programs in the language → writability
  • Cost of compiling programs → Ada compiling was very expensive which led to it not becoming a lot popular
  • Cost of executing programs → language design. Many run time checks prohibit fast code execution
    • Simple trade off can be made between compilation cost and execution task
    • Optimization is the name given to the collection of techniques that compilers use to decrease the size or increase the execution speed of the compiled code
    • In beginner orientation → no optimization required but in production where we compile once and use the compiled program later → optimization is needed
  • Cost of the language implementation system
    • Java had free compiler/interpreter systems
    • Expensive language or languages which only work on expensive systems not a good choice
  • Cost of poor reliability
    • Software shouldn’t fail in critical systems
  • Cost of maintaining programs → corrections and modifications to add new functionality → primarily on readability
  • Portability → ease with which programs can be moved from one implementation to another
    • BASIC not standardized
  • Generality → applicability to a wide range of applications
  • Well defined ness → completeness and precision of the languages official defining document

Language Design Tradeoffs

  • Java demands all references to array elements be checked but in C we don’t do index checking
  • So C programs tend to be faster than Java although java is more reliable
  • APL → large number of symbols → more expressivity → increased writability but less readability
  • The pointers of C++ can be manipulated in a variety of ways,
    which supports highly flexible addressing of data. Because of the potential reliability problems with pointers, they are not included in Java.

Implementation Methods

  • Language that a compiler translates is called the source language
  • Lexical analyzer → gathers the characters of the source program into lexical units
  • Lexical units are identifiers, special words, operators
  • Syntax analyzer takes the lexical units from lexical analyzer and uses them to construct parse trees
