Introduction

  • Programming languages support two sorts of abstractions
    • Data abstraction
    • Process abstraction
  • Subprograms are the form of process abstraction
  • They have a lot of general characteristics
    • Each subprogram has a single entry point
    • Caller is suspended when the subprogram is under execution
    • Control always returns to the caller once the called subprogram execution is completed
  • Subprogram definition describes the interface to and the actions of the subprogram abstraction
  • Subprogram call is the explicit request to execute a specific subprogram
  • Active subprogram is one which is called - begun to execute - and yet to complete the execution
  • Subprogram header is the first part of the definition and consists of the kind, name and the list of parameters
    • In C → void adder(parameter list)
    • In python → def adder(parameter list):
  • The parameter profile of a subprogram contains the number, order and the type of its formal parameters
  • The protocol of a subprogram is the parameter profile plus the return type of the subprogram

Parameters

  • Two ways a subprogram can gain access to data it is expected to process is
    • Non local variables → declared outside and visible in the subprogram
    • Through parameter passing
  • The data passed to the subprogram is accessed through names and are local to subprograms
  • Extensive access to non locals can reduce reliability

How methods are different?

  • Though methods access external data through non local references, the primary data to be processed by a method is the members of the object through which the method is invoked
  • Pure functional languages such as Haskell do not have mutable data, so functions cannot change memory in anyway
  • They simply perform computations and return the results

Categories of parameters

  • Formal parameters → appear in header and are thought of as dummy variables and these are bound to the storage only when the subprogram is called
  • Actual parameters is the list of parameters to be bound to formal parameters
  • Positional parameters is when binding between FP and AP is done by position, better when list is small
  • Keyword parameters is when the list is long, the programmer makes mistakes, to prevent this one solution is to provide keyword parameters, in which the name of the formal parameter to which the actual parameter is bound is specified in the function call, hence order is insignificant

Procedures and Functions

  • We have two distinct categories of subprograms
  • Procedures and functions, both are collections of statements and have parameters
  • Procedures do not return anything
  • Functions can return a value
  • Most languages do not include procedures
  • In place of a procedure, a function defined not to return value, can be used
  • The computations/statements of procedure can be enacted by a single call to the procedure
  • In ada, we have procedures and in fortran we have subroutines
  • Procedure can produce result in the calling program in two ways
    • Variables which are not local to subprograms, visible to both caller and the subprogram, both can change them
    • If the procedure has formal parameters that can transfer the data to the caller: Out parameter
  • Faithful functions will not modify its parameters nor modify outside data items, has no side effects and only compute some value and return it

Local referencing environments

  • Local variables: defined inside subprogram either in static or stack dynamic
  • Stack dynamic variables allow recursion
  • Cost of allocation, deallocation, initialization
  • Access is indirect and it’s not history sensitive
  • Static local variables have direct access and hence no run time overhead
  • History sensitive but can’t support recursion
  • In many PLs, they are stack dynamic unless declared static
  • Closure is RE + nested subprogram

Parameter Passing

  • Semantic Models of parameter passing
    • Receive data from corresponding actual parameter - IN
    • Transmit data to actual parameter - OUT
    • Both - INOUT
  • Implementations of semantic models of parameter passing

    Pass by value → IN mode semantics

    • Implemented by copy
    • C copied this from ALGOL 68
    • It’s faster but it makes a new copy which leads to use of storage

    Pass by result → OUT mode semantics

    • This method uses out-mode semantics. Just before control is transferred back to the caller, the value of the formal parameter is transmitted back to the actual parameter. This method is sometimes called call by the result.
    • In general, the pass-by-result technique is implemented by copying.

    Pass by value-result or Pass by Copy → INOUT mode semantics

    • This method uses in/out-mode semantics.
    • It is a combination of Pass-by-Value and Pass-by-Result. Just before the control is transferred back to the caller, the value of the formal parameter is transmitted back to the actual parameter. This method is sometimes called call by value-result.
    • the actual parameter is copied to the formal parameter at subprogram entry and then copied back at subprogram termination
    • It shares with pass-by-value and pass-by-result the disadvantages of requiring multiple storage for parameters and time for copying values.
    • It shares with pass-by-result the problems associated with the order in which actual parameters are assigned

    Pass by reference → INOUT mode

    • This transfers an access path and thus the calling subprogram is able to access the actual variable data path
    • Duplicate space is not required but it might be slower
    • One disadvantage is aliases are created

    Pass by name → INOUT mode

    • When parameters are passed by name, the actual parameter is, in effect, textually substituted for the corresponding formal parameter in all its occurrences in the subprogram.
    • This method is quite different from those discussed thus far; in which case, formal parameters are bound to actual values or addresses at the time of the subprogram call.
    • A pass-by-name formal parameter is bound to an access method at the time of the subprogram call, but the actual binding to a value or an address is delayed until the formal parameter is assigned or referenced

Overloaded subprograms

  • An overloaded operator is one that has multiple meanings
  • Meaning is determined by the type of operands
  • An overloaded subprogram is the one which has same name as another subprogram
  • Protocol must be unique even though name is same
  • Must differ in number of parameters, type and order
  • C++, Java, Ada, C# support method overloading
  • C++, Java, C# also support constructor overloading
  • In java, C# and C++ return type is not important
