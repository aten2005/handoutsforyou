OS Structures

Simple Structure

  • No well defined structure, and not divided into modules
  • A single crash can lead to full system failure
  • Apps are able to access basic IO to write directly to display and disk drives
  • Vulnerable to malicious programs, causing entire system crashes when user programs fail

UNIX Architecture

  • Systems Programs
  • Kernel
    • Consists of everything below the system call interface and above the physical interface
    • Provides file system, CPU scheduling
    • Large number of functions in one level
  • Full system crash due to kernel crash

Monolithic Kernel

  • Full OS in kernel Space
  • Very large kernel but faster
  • Hard to extend
  • If a service crashes, full kernel is affected
  • Very little overhead in system call interface or in communication within the kernel
  • Linux/Solaris → MS DOS

Microkernels

  • User services and kernel services are in separate address space
  • Smaller in size but slower
  • Extendible, all new services are added to user space
  • If a service crashes, system is not affected
  • More secure and reliable
  • A lot of overhead due to lack of availability of direct within kernel communications
  • Windows NT

Layered Approach

  • 0 → Bottom most layer → hardware
  • N → highest layer → UI
  • Every layer uses functions and services of the lower levels only
  • Also each layer hides the layers beneath it
  • We need to fix only the layers if we face any issues so reliable
  • While fixing also, we assume the lower levels are fixed and keep going up

Modules

  • Loadable kernel modules + core set of components in kernel
  • Links in additional services via modules, either at boot time or during run time
  • Each module has a well defined interface → object oriented
  • Dynamically linking services is preferable to adding new features directly to the kernel → does not require recompiling the kernel
  • Better than layered approach due to the fact that every module can call any module
  • Better than micro kernels due to the fact that there is no message passing overhead

OS Debugging

  • Debugging is finding and fixing errors and bugs
  • OS has a lot of log files
  • Failure of an app → core dump → memory of a process
  • OS Failure → crash dump → kernel memory

Performance Tuning

  • Improve performance by removing bottleneck → where it’s getting stuck
  • Trace listings of system activities, recorded for analysis
  • OS must provide means of computing and displaying measures for system → htop/task manager

System Boot

  • Program Counter(Register) stores the location of a boot program → loads kernel
  • POST → Power On Self Test → diagnostic that a BIOS runs to determine if hardware is working or not
  • If diagnostics pass, continues and executes boot block
    • First block on first track of a storage medium holds data used to start a system
    • Sometimes boot block is several physical sectors
    • Whichever disk/partition contains boot block is called boot disk/partition
    • Boot block contains remaining bootstrap program or address on disk and length of remainder of the bootstrap program
    • GRUB → Grand Unified Bootloader
    • After full bootstrap program it traverses the file system to locate the OS kernel and loads kernel into memory
  • POST/BIOS → bootloader → bootstrap program → OS kernel Image → loaded into main memory → fixed processes will start executing
  • UEFI → Unified Extensible Firmware Interface → Same as BIOS but gives some more benefits.
    • Data about init is stored in .efi file
    • Stored in efi system partition which also stores the bootloader
  • SWAP → Extra RAM that can be used in worst case scenarios. Store some extra processes in swap space
