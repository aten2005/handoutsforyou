Basics

  • Maximum CPU utilization obtained with multiprogramming
  • CPU - IO Burst Cycle → Process execution consists of a cycle of CPU execution and IO wait
  • CPU burst followed by IO burst
  • More number of short CPU bursts and less number of long CPU bursts

CPU Scheduler

Short Term/CPU Scheduler

  • Selects from among the processes in ready queue and allocates the CPU to one of them
  • CPU scheduling decisions may take place when a process
    • Switches from running to waiting state
    • Switches from running to ready state → interrupt
    • Switches from waiting to ready
    • Terminates

Preemptive Scheduling

  • Done in situations 2 and 3
  • Involuntary
  • Lets say we have a process P1 and it was supposed to execute only for 5s, then it’s paused and
    • [SITUATION 2] then P2 occurs
    • [SITUATION 3] other option is when P1 is busy, say P2 was doing something and when P1 gets ready again, P2 automatically is paused by force and P1 occurs

Non Preemptive Scheduling

  • Once a process is allocated the CPU, it retains the CPU until termination or switching to waiting state
  • Voluntary

Dispatcher

  • Dispatcher module gives control of the CPU to the process selected by the short term scheduler, this involves
    • switching context
    • switching to user mode
    • jumping to the proper location in the user program to restart that program
  • Dispatch latency → time it takes for the dispatcher to stop one process and start another running
  • The job of the dispatcher should be done very very fast

Scheduling Criteria

  • CPU Utilization → keep the CPU as busy as possible
  • Throughput → the number of processes that complete their execution per time unit → we need to maximize
  • Turnaround time → amount of time to execute a particular process, interval from submission time to completion time → we need to minimize
  • Waiting time → amount of time a process has been waiting in the ready queue → sum of durations, spent waiting to get into memory, waiting in ready queue, executing on CPU → we need to minimize
  • Response time → amount of time it takes from when a request was submitted until the first response is produced not output (for time sharing environment), depends on the speed of output device. Sort of like acknowledgement

Algo #2 - Shortest Job First (SJF) Scheduling

  • Associate with each process the length of it’s next CPU burst
  • Use these lengths to schedule the process with the shortest time
  • Use FCFS in case of tie
  • SJF is optimal → gives the minimum average waiting time for a given set of processes
  • The difficulty is knowing the length of the next CPU request
  • For long term scheduling in a batch system, use the process time limit that a user specifies when the job is submitted

Determining Length of Next CPU Burst

  • Not possible to implement for short term scheduling
  • Can only estimate the length and then we pick process with shortest predicted next CPU burst
  • Can be done by using length of previous CPU bursts using exponential averaging
    • t_n = actual length of nth CPU burst
    • Tou_n+1 = predicted value of the next CPU burst
    • 0 ≤ a ≤ 1 and Tou_0 is constant or overall system average
    • Tou_n+1 = at_n + (1-a)Tou_n

Prediction of Length of Next CPU Burst

  • if a = 0, tou_n+1 = tou_n
  • if a = 1, tou_n+1 = tn
  • tou_0 is constant or system average
  • Commonly we put a = 1/2
  • Can be non-preemptive or preemptive
  • The next CPU burst of a newly arrived process may be shorter than what is left of the currently executing process → happens in preemptive
  • Preemptive version is called shortest remaining time first → SRTF

Algo #3: Priority Scheduling

  • Priority number is associated with each process, generally starting from 0
  • The CPU is allocated to the process with the highest priority(smallest integer = higher priority)
  • Ties are broken using FCFS
  • SJF is priority scheduling where priority is the inverse of predicted next CPU burst time
  • Problem = Starvation/Indefinite blocking = low priority processes may never execute
  • Solution = Aging - as the time progresses, increase the priority of the process

Algo #4: Round Robin Scheduling

  • Each process gets a small unit of CPU time → time quantum q → aka time slice
  • After this time, the process is preempted and added to the end of the ready queue
  • If there are n processes in the ready queue and the time quantum is q, then each process gets 1/n of the CPU time in chunks of at most q time units at once
  • No process waits more than (n-1)q time units until it’s next time quantum
  • Timer interrupts every quantum to schedule next process
  • Performance
    • q very large → FCFS
    • q small → q must be large wrt to context switch, otherwise overhead is too high
  • Very important to draw the ready queue
  • We assume preemption happens even if there’s only one process

Algo #5: Multi level Queue Scheduling

  • Fix priority preemptive scheduling among queues
  • Reason for partitioning is that, we are going to classify the processes as well
  • For each queue, we can have multiple algorithms within themselves
  • But the con here is that there is no aging. Meaning this can lead to starvation
  • Processes in student processes may never get the chance to execute themselves
  • We can time slice among the queues → certain portion of the CPU time I will allocate to one o the queue

Algo #6: Multilevel Feedback Queue

  • A process can move between the various queues, aging can be implemented this way
  • Separate processes according to the characteristics of their CPU bursts
  • Multilevel feedback queue scheduler is defined by the following parameters
    • Number of queues
    • Scheduling algos for each queue
    • Method used to determine when to upgrade a process
    • Method used to determine when to demote a process
    • Method used to determine which queue a process will enter when that process needs service
  • NOTE: Process goes from head of Q0 to TAIL of Q1
  • As long as there is any process in Q0, no process from Q1 and Q2 get executed
  • The process will not execute anything in Q2 unless Q1 and Q0 are empty
  • If a process in Q2 is being executed and say we get a process in Q0, it gets preempted and Q0 executes first

Thread Scheduling

  • Distinction between user level and kernel level threads
  • User level threads are not scheduled
  • When threads supported, threads scheduled, not processes
  • Many to one and many to many models, thread library schedules user level threads to run on LWP (Kernel Level Resource)
    • Scheme is known as process contention scope since scheduling competition is within the process (local contention scope)
    • Typically done via priority set by programmer
  • Kernel thread scheduled onto available CPU is system-contention scope (SCS) - competition among all threads in system

P thread Scheduling

  • Allows specifying either PCS or SCS during thread creation, contention and scope values
  • PCS → PTHREAD_SCOPE_PROCESS
  • SCS → PTHREAD_SCOPE_SYSTEM
  • Can be limited by OS - Linux and Mac only allow SCS
  • On systems implementing M:M model, PCS policy schedules user level threads onto LWPs
  • SCS will create and bind an LWP for each user level thread effectively mapping a user level thread to a kernel level thread
  • pthread_attr_setscope(pthread_attr_t *attr, int scope)
  • pthread_attr_getscope(pthread_attr_t attr, int* scope)
  • On success, these return 0 and on error return non zero error number
  • 999 single threaded + 1 process with 1000 threads → you wouldn’t want the one process to starve out because of the others → very difficult

Completely Fair Scheduler - CFS

  • Linux is one to one mapping
  • Default scheduler in Linux kernel
  • Red Black Tree
  • Processes are given a fair amount of processor time
  • Keeps track of the amount of processor time provided to a process - virtual runtime
  • Smaller a process’s virtual runtime implies the process has been permitted to access processor for a smaller amount of time, hence higher is the need for processor
