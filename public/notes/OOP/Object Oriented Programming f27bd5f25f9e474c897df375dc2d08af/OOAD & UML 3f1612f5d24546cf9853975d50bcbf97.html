Decomposition

  • Algorithmic Decomposition is where the system is organized hierarchically in modules.
  • Each module denotes a significant step in some overall process, and the functionality process forms the basis of modularization.
  • OO Decomposition is where we view the world as a set of autonomous objects that collaborate to provide high-level functionality.
  • Each object is simply a conceptual or physical entity that exhibits some behavior.
  • AD highlights the ordering of events, while OOD emphasizes the objects and their actions and is more about how objects collaborate to provide functionality.
  • OOD is best suited for more complex systems, and here we can reuse the common mechanisms resulting in increased reusability

Object Oriented Analysis

  • Refers to the analysis of requirements from the point of view of classes and objects found in the vocabulary of the problem domain
  • Functional specs define the task to be performed readable by both domain expert and the SW developer
  • Functional Requirements define what is to be done and what is the functionality of the system
  • Non functional Requirements define how well the system provides functionality

Object Oriented Design

  • OOD leads to Object Oriented Decomposition
  • Uses the process of Object Oriented Decomposition and notation for depicting logical, physical and static, and dynamic models of the system
  • Models solution to the problem in terms of object
  • The goal is to identify the objects that the system contains,relationships and interactions between the objects

Example - Telephone

Problem Description

In a voice mail system, a person dials an extension number and, provided the other party does not pick up the telephone, leaves a message. The other party can later retrieve the messages, keep them, delete them. Real-world systems have a multitude of fancy features: messages can be forwarded to one or more mail boxes; distribution lists can be defined, retained and edited; and authorized persons can send broadcast messages to all users.

Use Cases

  • Use cases are an analysis technique to describe in a formal way how a system should work.
  • Each Use case focuses on describing the steps that are necessary to achieve a behavior and bring it to successful completion.
  • Each step represents an interaction with people or entities outside the system (actors) and the system itself
  • They describe a sequence of steps that yields a result
  • Leave a Message
    • Describe steps that caller must take to dial an extension and leave the message. Caller is the actor here
  • Example : Leave a message

CRC Cards

  • Class, Responsibilities, Collaborators
  • It’s an index card that describes a class it’s high level responsibilities and it’s collaborations i.e. dependent classes
  • It’s a design technique for discovering class, responsibilty and relationships
  • They are small and low tech and convenient to handle during brain storming sessions
  • Not a means of documenting design

Guidelines

  • Do not assign more than 3 responsibilities to a class
  • Maintain coherency and attach only related responsibilities
  • Don’t add unnecessary responsibilities
  • Delete classes with no responsibilities

Collaborations

  • Defines how objects collaborate to provide a functionality
  • Hence one class may participate in more than one collab

Interactions

  • OO Programs work by sending/receiving messages from/to objects
  • Message is a request by sender to execute an operation of a target object
  • This kind of message passing is called as interaction between objects

Design Patterns

Patterns

  • Means for capturing knowledge about problems and successful solutions in software development
  • Experience that has been gained in the past can be reused in similar situations
  • Reduces the effort in producing more resilient, more flexible and more efficient

Frameworks

  • Partially completed systems which contain unfinished elements which can be completed according to the requirement

Patterns vs Frameworks

  • Patterns are more abstract and general than frameworks
  • Pattern is a description of the way that a type problem can be solved but it itself is not a solution
  • Patterns are more primitive than frameworks
  • A framework can use several patterns but a pattern can’t employ a framework

Types of Design Patterns

Creational Pattern

  • Construction of Object instances
  • Separates the operation of an application from how it’s objects are created

Singleton Pattern

  • A singleton class has exactly one instance, and all clients need to access a single shared instance of a class
  • You want to ensure that no additional instances can be created accidentally
  • Define class with private constructor and supply a static method that returns a reference to the single object
public class DBManager 
{
	private static DBManager dbm=null;
	public static DBManager getInstance()
	{
		if(dbm==null)
		{
			dbm= new DBManager();
		}
		return dbm;
	}

	private DBManager() 
	{}
}

Structural Patterns

  • Concerned with the way the classes and objects are organized
  • Offer effective ways of using object oriented constructs such as inheritance, aggregation and composition

Behavioral Patterns

  • Address the problems that arise when assigning responsibilities to classes and when designing algorithms
  • Also focus on structural organization and also describe how objects communicate with each other
