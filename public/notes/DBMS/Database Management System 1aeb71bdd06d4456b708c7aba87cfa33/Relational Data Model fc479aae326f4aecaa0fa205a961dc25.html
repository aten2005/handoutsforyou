Relational Model

  • Database is a collection of relations
  • The notion of relation here is different from the notion of relationship used in ER modelling
  • Relation is the main construct for representing data in relational model
  • Every relation consists of a relation schema and relation instance

Relation Schema

  • Denoted by R=(A1, A2, A3, A4, … An)
  • The number of columns in a relation is known as it’s arity
  • Number of entities in a relation is it’s degree
  • The number of tuples is the cardinality 🙂

Relation State r of R

  • Also known as the table
  • Each row in table represents a collection of related data
  • Each row contains facts about some entity of some entity set
  • r(R) is a set of n tuples in R where r = {t1,t2,t3…tnt1, t2, t3… tnt1,t2,t3…tn﻿} and R = (A1, A2, A3, … An)
  • r is an instance of R each t is a tuple and is a ordered list of values
  • t = (v1,v2…vnv1, v2 … vnv1,v2…vn﻿) where vi is an element of domain of Ai
  • Entities of each type/set are stored as rows in a single relation
  • Hence in general, a relation corresponds to a single entity type in ER diagram
  • In some cases, a relationship between two entities can have some specific attributes which can be captured in a relation(table)
  • A row is called a tuple. The columns of the table represent attributes of that entity type
  • The column header is known as attribute or field
  • Data type or format of an attribute is the format of data for that attribute
  • Values that can appear in a column for any row is called the domain of that attribute

Relational Database Schema

Example: Student is the relation name and Student.age denotes the age attribute of student relation

Relational Model Terms

  1. Relation is Table
  1. Attribute is Column Header
  1. Domain is all possible column values
  1. Tuple is Row
  1. Schema of a relation is a table definition
  1. State of a relation is the populated table

Relational Constraints

  1. Domain Level Constraints → Format of Data, Semantic i.e. NOT NULL
  1. Entity Integrity constraints → Primary Key, Unique key
  1. Referential Integrity Constraints → Foreign Key
  1. Dependencies → What attributes value defines the value of another attribute is known as a dependency
  1. This concept is used in database design

Referential Integrity

  • Specified between two relations and is used to maintain consistency among the tuples of two relations
  • This helps in maintaining consistency because, instead of having a foreign key, we have one full table for storing data of 2 tables combined, it might lead to issues like
    1. Data Redundancy
    1. Data Inconsistency: if a column is edited in the main table but not in the combined table
  • The value of FK in a tuple t of R1, either occurs under PK in R2 for some tuple or is null
  • R1 is the referencing relation → Foreign Key
  • R2 is the referenced relation → Primary key
  • Actions need to be taken when an FK is set up like for insert, update and delete

Super Key

  • Super Key → All columns which are unique if put together in a table is a super key
  • Minimal Super Keys are Candidate keys
  • One of the candidate keys gets chosen as Primary Key
  • Other candidate keys become alternate key

Super Key Textbook Definition

  • In Simple, if we have a table with ABCDEF and A/B/D are candidate keys, then any one of them can become the primary key, and then the super key will include every combination which has A/B/D in them.
