Introduction

  • Database is a collection of related data where data means known facts that can be recorded and have implicit meaning
  • Database is also relatable to mini world or universe of discourse in real life
  • A database is a logically coherent collection of data with some inherent meaning
  • A database is designed, built, and populated with data for a specific purpose
  • A database management system is a computerized system that enables users to create and maintain a database
  • It is a general purpose software system that facilitates the processes of defining, constructing, manipulating and sharing the databases among various users and applications
    • Defining a database involves specifying the data types, structures, and constraints of the data to be stored in the database. The database definition or descriptive information is also stored by the DBMS in the form of a database catalog or dictionary; it is called meta-data.
    • Constructing a database is the process of storing the data on some storage medium that is controlled by the DBMS.
    • Manipulating a database includes functions such as querying the database to retrieve specific data, updating the database to reflect changes in the mini world, and generating reports from the data.
    • Sharing a database allows multiple users and programs to access the database simultaneously
    • Protecting a database includes system protection against hardware and software malfunction and security protection against malicious usage
  • Database + DBMS = Database System

Example

  • We consider an university database
  • Student file stores data on each student
  • Course file stores data on each course
  • Section on each section and grade report stores grades and prerequisites stores the same
  • We need to specify different types of data elements

Characteristics of Database Approach

Self describing nature

  • Database system contains not only the database but also the definition and description of the database structure and constraints
  • This definition is stored in the DBMS catalog which contains information such as the structure of each file, the type and storage format of each data item and various constraints on the data
  • The information stored in the catalog is called metadata and it describes the structure of the primary database
  • In NoSQL DB(s) everything is self describing

Insulation between programs and data → Data abstraction

  • Traditionally structure of datafiles is embedded in applications and any changes to the structure of a file may require changing all programs that access that file
  • In contrast, DBMS access programs do not require such changes in most cases.
  • The structure of data files is stored in the DBMS catalog separately from the programs
  • We call this property PROGRAM-DATA INDEPENDENCE
  • For example, if we add a column into an application, then we have to change the application as well, but in DBMS only the schema can be changed
  • An operation on data is defined in two parts
  • The interface(or signature) of an operation includes the operation name and the data types of its arguments (or parameters).
  • The implementation (or method) of the operation is specified separately and can be changed without affecting the interface.
  • User application programs can operate on the data by invoking these operations through their names and arguments, regardless of how the operations are implemented. This may be termed program-operation independence.
  • The characteristic that allows program-data independence and program-operation independence is called data abstraction

Support of Multiple views of data

  • View maybe a subset of the data or may contain virtual data that is derived but is not explicitly stored

Sharing of data and multiuser transaction processing

  • Database software must include concurrency control so that if multiple people are editing the data at the same time, these edits are taken in a controlled manner
  • The concept of transaction is central to many database applications
  • A transaction is an executing program or process that includes one or more database accesses such as reading or updating of database records
  • Each transaction is supposed to execute a logically correct database access if executed in its entirety without interference from other transactions. T
  • The DBMS must enforce several transaction properties.
    • The isolation property ensures that each transaction appears to execute in isolation from other transactions, even though hundreds of transactions may be executing concurrently.
    • The atomicity property ensures that either all the database operations in a transaction are executed or none are.

Disadvantages of Traditional Storage

  • Data redundancy leading to inconsistency
  • Data is scattered across different files
  • Integrity problems
  • No Atomicity due to no transaction management
  • Concurrency control is not there
  • Security provided by OS is not adequate

Advantages of using DBMS

  • Give access control privileges
  • Persistent storage
  • Multiuser interface
  • Integrity constraints can be specified
  • Efficient backup and recovery methods
  • Data independence

Disadvantages of DBMS

  • Extra cost for software, hardware and training
  • Not suitable or effective for certain applications
  • Data manipulation not supported by query languages

Data Model

  • ER model has no concept of Primary Keys while relational model does

High Level Data Model

  • Descriptions are close to user views
  • Conceptual models
  • We use concepts like entity, attributes and relationships

Low Level Data Model

  • Give details about how data is stored in computers
  • Storage level details
  • Indexes, Clusters, Files

Representational or Implementation Data Models

  • Between high and low
  • We use networks, objects, trees and tables to describe data
  • Ideally every column should contain only non complex types and single valued

Network Model

  • Relationship is captured using links
  • Integrated Data Store - IDS is an example
  • In network model, a relationship between customers and accounts is shown in the form of network and not tables

Hierarchical Model

  • Data organized as collection of trees
  • IMS - Information Management System from IBM (Remember as HIMs)
  • Here we have any account and then under that account we have users as a hierarchy
  • Here we have a lot of redundancy

Relational Model

  • Data are captured as tables and keys
  • MySQL, Oracle etc.
  • Example can be customers and accounts
  • Here one customer can have multiple accounts so we use joins and try to get the answer for which customer has which account or which account has which owner

Object Data Model

  • Objects created through Object oriented programming can be stored in database

Object Relational Model

  • Objects stored in tables and has relations too
  • ORDBMS

Database Schema

  • Description of a database is called database schema
  • A database can be described using 3 levels of abstractions

1. External Schema

  • Used to describe database at an external level
  • Data access to be customized in level of individual users/groups and applications
  • End user requirements

2. Conceptual Schema(Logical Schema)

  • Is only one not multiple
  • Describes all relations in RDBMS
  • Describes stored data in DBMS
  • Need to arrive at good choice of relations, fields and constraints

3. Physical Schema

  • Is only one not multiple
  • Describes the physical storage strategy for the database

Data Independence

Logical Data Independence

  • Changes in conceptual level schema should not affect the application level or external level schemas
  • For example: consider two users A & B. Both are selecting the fields "Employee Number" and "Employee Name".
  • If user B adds a new column (e.g. salary) to his table, it will not affect the external view for user A, though the internal schema of the database has been changed for both users A & B.

Physical Data Independence

  • The changes in physical features of storage i.e. changes to the physical storage format should not affect schema at conceptual level
  • Change internal schema without changing conceptual schema
