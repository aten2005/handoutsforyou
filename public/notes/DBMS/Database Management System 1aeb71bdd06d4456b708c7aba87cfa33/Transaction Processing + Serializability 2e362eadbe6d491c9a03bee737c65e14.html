Introduction

  • We have railway reservation, and we have to reserve x number of berths
  • Process is the abstraction for a running program
  • A Transaction is an executing program that forms a logical unit of database processing.
  • A transaction includes one or more database access operations – read/write/delete etc.
  • A transaction is an atomic unit of work that is either completed in its entirety or not done at all.
  • For recovery purposes, the system needs to keep track of when the transaction starts, terminates, and commits or aborts.

ACID Properties

  • Atomicity: A transaction is an atomic unit of processing. It is either performed in its entirety or not performed at all.
  • Consistency Preservation: The successful execution of a transaction takes the database from one consistent state to another.
  • Isolation: A transaction should be executed as if it is not interfered by any other transaction.
  • Durability: The changes applied to the data by a transaction must be permanent.

Transaction States

  • Active State: Initial state when a transaction starts.
  • Partially Committed State: This state is reached when the last statement is executed and the outcome is not written to the DB. Here all the statements are executed but the data is not permanently written onto the DB
  • Failed State: After discovering that the normal execution cannot be continued a transaction is aborted and reaches failed state.
  • Committed State: Is reached after successful completion of the transaction.
  • Terminated State: Is reached after failure or success of the transaction

Transaction Scheduling and Concurrency

  • The descriptions that specify the execution sequence of instructions in a set of transactions are called as schedules.
  • Hence schedule can describe the execution sequence of more than one transaction.
  • Here, in the above schedule the transactions T1 & T2 are executed in a serial manner i.e., first all the instructions of the transaction T1 are executed, and then the instructions of T2 are executed. Hence the above schedule is known as serial schedule.
  • In serial schedule, we have less performance but more consistency
  • Serial Schedule does not exploit concurrency
  • If transactions, are concurrent then resources can be used efficiently
  • Concurrent schedules might result in incorrect database state

Problems with Concurrent Scheduling

  • Lost Update Problem → the value is changed by second transaction before it’s used in T1
  • Temporary Update Problem → when one transaction fails and so the update is reverted
  • Incorrect Summary Problem
  • Correctness > Speed and efficiency

Conflicting Operations

  • Must belong to two different transactions
  • Must access same data item
  • One of them must be write operation

Conflicting Equivalent Schedules

  • If a schedule S can be transformed into a schedule Sˈ by a series of swaps of non-conflicting instructions, we say that S and Sˈ are conflict equivalent.
  • Further, we say that a schedule S is conflict serializable if it is conflict equivalent to a serial schedule.
  • In our example S4 in the above example is a serial schedule and is conflict equivalent to S1. Hence S1 is a conflict serializable schedule

Conflict Serializable Schedules

  • We only swap non conflicting operations

Test for Conflict Serializability

  • We construct a Precedence Graph
  • Each transaction participating in the schedule will become a vertex
  • The set of edges consist of all edges Ti → Tj for which one of the following conditions holds
    • Ti executes W(Q) before Tj executes R(Q)
    • Ti executes R(Q) before Tj executes W(Q)
    • Ti executes W (Q) before Tj executes W(Q)
  • Note that R(Q) before W(Q) does not count

Types of Schedules

Serial schedule:

  • A schedule S is serial if, for every transaction T participating in the schedule, all the operations of T are executed consecutively in the schedule.
  • Otherwise, the schedule is called non serial schedule.

Serializable schedule:

  • A schedule S is serializable if it is equivalent to some serial schedule of the same n transactions
  • Being serializable is not the same as being serial
  • Being serializable implies that the schedule is a correct schedule.
  • It will leave the database in a consistent state.
  • The interleaving is appropriate and will result in a state as if the transactions were serially executed, yet will achieve efficiency due to concurrent execution

View Serializability

Let S and Sˈ be two schedules with the same set of transactions. S and Sˈ are view equivalent if the following three conditions are met, for each data item Q,

  1. If in schedule S, transaction Ti reads the initial value of Q, then in schedule Sˈ also transaction Ti must read the initial value of Q. → In both schedules, first read should be in same transaction from the storage
  1. If in schedule S transaction Ti executes read(Q), and that value was produced by transaction Tj (if any), then in schedule Sˈ also transaction Ti must read the value of Q that was produced by the same write(Q) operation of transaction Tj
  1. The transaction (if any) that performs the final write(Q) operation in schedule S must also perform the final write(Q) operation in schedule S → In both schedules, last write should be in same transaction

A Schedule S is view serializable if it is view equivalent to a serial schedule.

  • Every conflict serializable schedule is also view serializable

Recoverable Schedules

  • It is where if a transaction Tj reads a data item previously written by a transaction Ti then the commit operation of Ti appears before the commit operation of Tj
  • The following is not recoverable if T9 commits immediately after the read
  • If T8 should abort, T9 would have read an inconsistent database state. Hence database must ensure that schedules are recoverable.
  • Point being since T8 is writing and say it fails, T9 will enter inconsistent state
  • A schedule S is recoverable if no transaction T commits until all transactions T’ that have written an item that T reads has committed

Cascading Rollbacks

  • A single transaction failure leads to a series of transaction rollbacks
  • Consider the following schedule where none of the transactions has yet committed (so the schedule is recoverable)
  • So if there is no commit it’s still recoverable?
  • Here if T10 fails, T11 and T12 must also be rolled back and this can lead to undoing of a significant amount of work
  • This is called cascading rollbacks

Cascade less Schedules

  • Cascading rollbacks cannot occur
  • For each pair of transactions Ti and Tj such that Tj reads a data item previously written by Ti, the commit operation of Ti appears before the read operation of Tj
  • Every cascade less schedule is also recoverable
  • It is desirable to restrict the schedules to those that are cascade less

Examples

{r1(X); r2(X); w1(X); r1(Y); w2(X); c2; w1(Y); c1} = Is recoverable

  • r1(X) and r2(X) both read X, but neither writes to it, so there is no conflict.
  • w1(X) writes to X, but it does not conflict with the previous reads because they have already completed.
  • r1(Y) reads Y, but no other transaction has written to Y yet.
  • w2(X) writes to X, but it does not conflict with the previous reads because they have already completed.
  • c2 commits, which means that w2(X) is now visible to other transactions.
  • w1(Y) writes to Y, but it does not conflict with the previous reads because they have already completed.
  • c1 commits, which means that w1(Y) is now visible to other transactions.
